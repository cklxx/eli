# 2026-04-10 · Repository Review

## Conclusion

`eli` already has the core of a strong AI project.

The architecture is clear, the product direction is differentiated, and the repository is unusually friendly to agent-assisted development. The project can grow into a top-tier AI framework.

It is **not yet ready** for large-scale maintainer growth.

The limiting factors are no longer the core runtime. They are the support systems around it:

1. documentation trust
2. cross-language CI coverage
3. local workflow packaging
4. maintainer governance
5. extension contract clarity

Today, `eli` is best described as a **high-quality author-driven system**. The next step is to turn it into a **collaboration-ready platform**.

---

## Direct Answers

### Can this become a top-tier AI project?

Yes.

Why:

- the project has a clear thesis: hook-first, Rust-native, multi-channel AI runtime
- the codebase has real internal structure across `eli`, `nexil`, and `sidecar`
- the repository already contains agent-native operating guidance
- the system shows signs of real operational learning, not just feature accumulation

Still missing:

- stronger public docs as source of truth
- system-shaped CI
- stable extension contracts
- maintainer-facing governance

### Can this support many maintainers?

Not safely yet, but it can.

Why not yet:

- contributors still need too much implicit context
- docs are not uniformly trustworthy
- CI does not fully reflect the actual system shape
- some key extension surfaces are not yet hardened as long-lived contracts

What would change that:

- clear contribution workflow
- ownership and review routing
- contract docs for hooks, tools, and sidecar integration
- compatibility expectations and architecture decision records

## Main Gaps

## P0

### P0. Public documentation has been allowed to drift from the implementation

Evidence:

- `README.md` had stale crate versions
- older docs in `docs/` described older package names and runtime shapes

Why it matters:

- public docs are the first contract most contributors see
- once the entry docs are unreliable, all later onboarding becomes slower and riskier

What to do:

- keep `README.md` and `docs/index.md` aligned with current code
- clearly mark historical review documents as snapshots

---

## P1

### P1. CI has been narrower than the real system

Evidence:

- CI covered Rust checks well
- but the repo also contains Python tests and a TypeScript sidecar

Why it matters:

- the actual product is multi-language
- regressions in gateway, Python harness, or sidecar can bypass Rust-only CI

What to do:

- keep Rust checks fast and strict
- add Python smoke validation
- add sidecar test/build validation
- reserve heavy integration for scheduled or gated lanes

### P1. The outer architecture is cleaner than the inner execution boundary

The repository presents a hook-first runtime, but important execution behavior still lives behind an inner `Agent` loop.

Evidence:

- outer turn pipeline in `crates/eli/src/framework.rs`
- model execution delegation in `crates/eli/src/builtin/mod.rs`

Why it matters:

- observability is weaker than the public architecture suggests
- future policy injection becomes harder
- long-term extensibility depends on better visibility around tool-loop events

What to do:

- keep the current design for now
- treat this as an explicit architecture boundary to improve later
- add finer-grained lifecycle hooks around tool-loop behavior when the next runtime pass happens

---

## P2

### P2. Local workflow packaging was too thin for the real project

Evidence:

- the repo spans Rust, Python, and TypeScript
- but local workflow entrypoints were much thinner than the actual maintenance burden

Why it matters:

- contributor quality drops when the correct command path is unclear
- agent workflows become less repeatable

What to do:

- keep a richer `justfile`
- move real logic into `scripts/`
- maintain stable command contracts for common tasks

### P2. Maintainer governance is still lighter than the project ambition

The repository has strong local style and operating rules, but many-maintainer scale needs more than that.

Why it matters:

- large maintainer surfaces need explicit workflow, ownership, and review routing
- otherwise architecture quality depends too heavily on author context

What to do:

- maintain a current `CONTRIBUTING.md`
- add review and ownership surfaces such as `CODEOWNERS`
- define architectural review expectations for contract-sensitive changes

### P2. Extension surfaces are real, but not yet platform-grade contracts

The project already exposes multiple extension points:

- hooks
- tools
- skills
- channels
- sidecar integration

Why it matters:

- frameworks scale through stable contracts, not only internal quality
- many maintainers and external contributors need explicit compatibility boundaries

What to do:

- document hook semantics
- document tool naming and result conventions
- document sidecar contract expectations
- define what counts as a breaking change

### P2. A few structural pressure points should be watched carefully

The project does not need a rewrite, but it does have a few pressure points:

- global tool registry state
- `BuiltinImpl` carrying many runtime concerns
- the gap between hook architecture and inner tool execution

These are not immediate failures. They are places where complexity can keep growing if left unmanaged.

---

## Priority Sequence

The next maturity step is not a sweeping rewrite.

It is to harden the repository around the core:

1. current docs that can be trusted
2. CI that matches the real system
3. stable local workflow commands
4. maintainer-facing contribution rules
5. extension contracts that are explicit and durable

---

## Final Judgment

`eli` has the architecture and identity required to become a top-tier AI project.

Its biggest challenge is no longer invention. It is maturity.

If the project continues to improve documentation, CI, workflow packaging, governance, and extension contracts, it can scale beyond an author-centric system and become a serious many-maintainer platform.
